# PROJECT.md — «Красавица»

## 1. Краткое описание проекта

«Красавица» — Telegram-бот / AI-помощник для подбора косметического, не лекарственного ухода за кожей. Пользователь описывает проблему кожи, тип кожи, бюджет или желаемое средство, а бот формирует базовый уход, предлагает конкретные варианты косметики, даёт ссылки на маркетплейсы и собирает обратную связь по рекомендациям.

Текущий продуктовый вектор: перейти от простого AI-бота к более универсальной beauty-платформе, которая постепенно сможет подбирать не только уходовую косметику, но и парфюм, сохраняя премиальный минималистичный UX.

## 2. Текущий репозиторий и инфраструктура

Репозиторий:
https://github.com/DrDrozdov/krasavitsa_bot

Основные файлы проекта:

- `bot.py` — основной Telegram-бот, обработчики сообщений, кнопки, сценарии, карточки товаров, feedback, admin stats.
- `ai_client.py` — запросы к AI Tunnel / DeepSeek-compatible API.
- `prompts.py` — системный промпт для AI-подбора косметического ухода.
- `links.py` — генерация поисковых ссылок на маркетплейсы.
- `database.py` — SQLite-база, пользователи, история рекомендаций, feedback, рейтинги средств, admin stats.
- `requirements.txt` — зависимости.
- `Procfile` — запуск worker на Railway.
- `.gitignore` — исключение `.env`, `venv`, `__pycache__`, `*.db`.

Деплой:
- Railway.
- Запуск через `Procfile`.
- Переменные окружения хранятся в Railway Variables.

Переменные окружения:
- `BOT_TOKEN`
- `DEEPSEEK_API_KEY`
- `DEEPSEEK_MODEL`
- `ADMIN_IDS` — список ID администраторов через запятую.

Важно:
- Не хардкодить токены.
- Не коммитить `.env`.
- Не коммитить `krasavitsa.db` и любые `.db` файлы.
- Использовать только одну основную SQLite-базу: `krasavitsa.db`.

## 3. Текущее состояние функций

### 3.1. Основной AI-подбор ухода

Бот принимает свободный текст пользователя и через AI формирует JSON-ответ со структурой:

- `summary` — что понял по запросу;
- `morning` — утренний уход;
- `evening` — вечерний уход;
- `recommended_products` — конкретные средства;
- `search_queries` — что искать;
- `avoid` — чего избегать;
- `warning` — когда лучше обратиться к врачу.

Системный промпт запрещает:
- ставить диагнозы;
- рекомендовать лекарства, антибиотики, гормональные мази и лечение;
- давать медицинские назначения.

Бот должен оставаться именно помощником по косметическому, не лекарственному уходу.

---

### 3.2. Главное меню

Реализованы кнопки:

- «Подобрать уход»
- «Мой тип кожи»
- «Бюджет»
- «Где искать»
- «Мои подборы»
- «Когда к врачу»
- «Помощь»

---

### 3.3. Быстрые сценарии

Реализованы сценарии:

- сухость;
- жирный блеск;
- чувствительность;
- акне;
- SPF-защита;
- другое.

При нажатии бот формирует готовый смысловой запрос и отправляет его в основную логику подбора.

---

### 3.4. Бюджет

Реализованы кнопки бюджета:

- до 1000 ₽;
- до 3000 ₽;
- до 6000 ₽;
- без ограничений.

Бюджет сохраняется в профиль пользователя через `update_user_profile`.

---

### 3.5. Карточки товаров

Для каждого рекомендованного средства бот формирует отдельную карточку:

- эмодзи по категории средства;
- название средства в формате monospace / copy-friendly;
- рейтинг средства;
- кнопки маркетплейсов;
- кнопки оценки конкретного средства.

Ссылки формируются на:

- Яндекс Маркет;
- Ozon;
- Wildberries;
- Золотое Яблоко;
- Лэтуаль;
- Рив Гош.

Сейчас ссылки ведут на поисковую выдачу по названию товара, а не на точные карточки товара.

---

### 3.6. Feedback по подбору

После общего ответа бот предлагает оценить подбор:

- «Полезно»
- «Не подошло»

Оценка сохраняется в таблицу `recommendations`.

---

### 3.7. Feedback по конкретным средствам

Под каждой карточкой средства есть кнопки:

- «Подходит»
- «Не подходит»

Оценка сохраняется в `product_feedback`.

Для безопасной работы callback используется `product_id`, а не длинное название товара.

---

### 3.8. Рейтинг средств

Реализована логика рейтинга:

- `recommended_products` хранит выданные пользователю средства;
- `product_feedback` хранит оценки средств;
- `normalize_product_name()` формирует `product_key`;
- статистика группируется по `product_key`;
- `get_product_rating()` выводит процент положительных оценок, likes/dislikes и статус «Новинка», если оценок ещё нет.

Цель этой системы — постепенно собирать реальные данные о том, какие средства пользователи считают подходящими.

---

### 3.9. История подборов

Реализована кнопка «Мои подборы».

Бот показывает последние запросы пользователя, дату и статус оценки.

---

### 3.10. Админ-статистика

Реализована команда `/admin_stats`.

Доступ только для пользователей из `ADMIN_IDS`.

Показывает:

- количество пользователей;
- количество рекомендаций;
- статистику оценок подборов;
- статистику оценок товаров;
- топ средств.

Также есть команда `/product_stats`, которая показывает статистику по средствам.

## 4. Важные технические проблемы и слабые места

### 4.1. Форматирование файлов

В актуальном GitHub raw-файлы выглядят как сильно сжатые / однострочные файлы. Это не всегда мешает Python, если синтаксис валиден, но резко ухудшает читаемость и риск правок.

Приоритет:

- привести `bot.py`, `database.py`, `ai_client.py`, `links.py`, `prompts.py` к нормальному многострочному форматированию;
- прогнать форматирование через Black;
- проверить синтаксис перед деплоем.

### 4.2. Debug-логи

В `bot.py` сейчас есть debug-вывод переменных окружения:

- `ALL ENV KEYS`
- `BOT_TOKEN DEBUG`

Это опасно и визуально загрязняет логи.

Нужно удалить вывод токена и любых чувствительных данных.

### 4.3. HTML/Markdown-разметка

В карточке товара важно использовать корректный формат:

- если `parse_mode="HTML"`, то название должно быть `<code>{product_name}</code>`;
- если используется Markdown, то обратные кавычки.

Нельзя смешивать Markdown-разметку с `parse_mode="HTML"`.

### 4.4. Точность товарных ссылок

Сейчас ссылки ведут на поиск, а не на конкретные карточки товара.

Это нормально для MVP, но ограничивает доверие к подбору. В будущем нужен каталог или интеграция с товарными API/ручными ссылками.

### 4.5. Стабильность рекомендаций

AI может каждый раз выдавать немного разные названия средств. Для рейтинга это частично решено через `product_key`, но для точных карточек и ссылок нужна более строгая нормализация или каталог.

## 5. Продуктовое позиционирование

Текущее позиционирование:

«Красавица» — AI-помощник, который понимает кожу и подбирает уход.

Тон:

- минималистичный;
- премиальный;
- спокойный;
- понятный;
- без медицинских обещаний;
- без агрессивного маркетинга;
- ближе к Apple / Notion / modern beauty-tech.

Не использовать стиль:

- «чудо-бот»;
- «лечит акне»;
- «гарантирует результат»;
- «самый лучший уход»;
- «революционный ИИ».

## 6. Возможное расширение: парфюм

Есть идея расширить проект из skincare-бота в более универсальную beauty-платформу.

Потенциальное направление:

- подбор ухода за кожей;
- подбор косметики;
- подбор парфюма;
- подбор подарков в beauty-категории.

Парфюм можно добавить как отдельный режим:

- «Подобрать уход»
- «Подобрать парфюм»

Для парфюма бот может спрашивать:

- пол;
- возраст / стиль;
- сезон;
- повод;
- бюджет;
- любимые ноты;
- что не нравится;
- примеры ароматов, которые уже нравятся.

Важно:

- не смешивать skincare и parfum в одном промпте без маршрутизации;
- сначала добавить отдельный режим;
- потом сделать отдельный `perfume_prompt`;
- потом отдельную структуру ответа и отдельную статистику.

1. В истории показывать не только запросы, но и краткую рекомендацию.

4. Использовать сохранённый профиль пользователя в новых запросах.
5. Добавить команду «Мой профиль».
6. Добавить редактирование профиля:
   - тип кожи;
   - бюджет;
   - чувствительность;
   - любимые бренды;
   - нежелательные бренды.

### Этап 5. Точность товаров и ссылок

1. Оставить поиск как fallback.
2. Создать небольшой curated-каталог популярных средств.
3. Для каталога хранить:
   - canonical_name;
   - product_key;
   - category;
   - skin_type;
   - budget;
   - image_url;
   - marketplace links;
   - tags.
4. Если AI предлагает товар из каталога — показывать точные карточки.
5. Если товара нет в каталоге — использовать поисковые ссылки.
6. Постепенно пополнять каталог на основе популярных запросов.

### Этап 6. Парфюм как второй модуль

1. Добавить кнопку «Подобрать парфюм».
2. Создать отдельный prompt для парфюма.
3. Сделать отдельную структуру JSON:
   - style;
   - occasion;
   - season;
   - notes;
   - recommended_perfumes;
   - avoid;
   - search_queries.
4. Сделать карточки ароматов:
   - название;
   - ноты;
   - сезон;
   - стиль;
   - рейтинг;
   - ссылки.
5. Добавить отдельную статистику по ароматам.
6. Не ломать skincare-логику.
